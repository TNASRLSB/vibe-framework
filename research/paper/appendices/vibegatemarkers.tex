\section{VIBE\_GATE Marker Specification}
\label{app:vibegatemarkers}

Each VIBE Framework skill defines a set of verification markers that must be
emitted via \texttt{echo "VIBE\_GATE: key=value"} in a Bash tool call before
the model may claim completion. The value after \texttt{=} must come from a
command substitution---never hardcoded---so the sentinel can trust that the
value reflects actual file-system or tool state.

Table~\ref{tab:gate-markers} lists all 35 markers identified across the five
skill files examined. The original task specification anticipated 37 markers;
the discrepancy of 2 may reflect markers defined in skills not included in this
inventory (e.g., Heimdall, Orson, Forge, Scribe) or a different counting
methodology for composite markers.

\paragraph{Marker semantics.}
Markers typed as ``boolean'' emit 1 or 0. Markers typed as ``integer'' emit
a non-negative count. The sentinel's Check~6 cross-references integer markers
against numerical claims in the model's completion message: if the message
claims a number $N > 5$ and a marker shows a value less than $N$, the check
fails with high confidence.

\paragraph{Self-reported markers.}
Two of Emmet's markers (\texttt{red\_phase\_confirmed} and
\texttt{green\_phase\_confirmed}) are self-reported: the model emits the
value~1 after asserting it performed the red-green validation cycle. These
cannot be verified mechanically after the fix is restored. The sentinel's
Check~3 (test claim without execution) provides partial coverage by verifying
that test commands were actually run via Bash during the turn.

\paragraph{Coverage.}
The 35~markers span four domain skills and one shared protocol. Skills not
inventoried here (Heimdall, Orson, Forge, Scribe) may define additional
markers. The architecture is extensible: any skill can define new
\texttt{VIBE\_GATE} markers and the sentinel will automatically detect them
without code changes.
