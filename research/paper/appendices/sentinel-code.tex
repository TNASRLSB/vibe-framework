\section{Completion Sentinel Source Code}
\label{app:sentinel}

This appendix presents the mechanically interesting portions of the completion
sentinel hook (\texttt{scripts/completion-sentinel.sh}), which implements
Layer~1 of the Completion Integrity System. The script executes on every
\texttt{Stop} and \texttt{SubagentStop} hook event, parsing the session
transcript to detect discrepancies between the model's completion claims
and the observable tool-use evidence.

\subsection{Transcript Parser}

The sentinel's first phase extracts structured data from the last
conversational turn using a single \texttt{jq} pipeline. It identifies the
last genuine user message (excluding tool-result-only messages), then counts
all tool calls, categorizes them by type, extracts file read targets,
collects \texttt{VIBE\_GATE} markers from Bash outputs, and checks whether
the model performed any verification after dispatching subagents.

\begin{lstlisting}[language=bash,caption={Phase A: Transcript extraction via jq.}]
TURN_DATA=$(jq -s '
  # Find last user message with actual text content
  # (not just tool_results)
  (
    [to_entries[] | select(
      .value.type == "user" and (
        (.value.message.content | type) == "string" or
        ((.value.message.content | type) == "array" and
          (.value.message.content
            | map(select(.type == "text"))
            | length > 0) and
          (.value.message.content
            | map(select(.type == "tool_result"))
            | length == 0))
      )
    )] | last.key // 0
  ) as $last_user_idx |

  # All assistant tool_use blocks after that index
  [.[$last_user_idx:][] |
    select(.type == "assistant") |
    .message.content[]? |
    select(.type == "tool_use") |
    { name: .name, input: .input }
  ] as $tool_calls |

  # All tool_result text content after that index
  [.[$last_user_idx:][] |
    select(.type == "user") |
    .message.content[]? // empty |
    select(type == "object"
      and .type == "tool_result") |
    (.content[]? // empty
      | select(type == "object"
        and .type == "text") | .text) // ""
  ] as $tool_results |

  # Count by tool name
  ($tool_calls | group_by(.name)
    | map({key: .[0].name, value: length})
    | from_entries) as $counts |

  # VIBE_GATE markers from tool results
  [$tool_results[]
    | capture("VIBE_GATE: (?<key>[^=]+)=(?<val>.+)")
    // empty
  ] as $gate_markers |

  # Check if there are Read/Bash calls AFTER
  # the last Agent call
  (
    if ([$tool_calls[]
      | select(.name == "Agent")] | length) > 0
    then
      ($tool_calls | to_entries
        | map(select(.value.name == "Agent"))
        | last.key) as $last_agent_idx |
      [$tool_calls[$last_agent_idx + 1:][]
        | select(.name == "Read"
          or .name == "Bash")]
        | length > 0
    else true end
  ) as $verified_after_agent |

  {
    tool_counts: $counts,
    total_tools: ($tool_calls | length),
    gate_markers: $gate_markers,
    has_skill: (([$tool_calls[]
      | select(.name == "Skill")] | length) > 0),
    verified_after_agent: $verified_after_agent,
    has_agent_calls: (([$tool_calls[]
      | select(.name == "Agent")] | length) > 0)
  }
' "$TRANSCRIPT" 2>/dev/null)
\end{lstlisting}

\subsection{Six Independent Checks}

The sentinel runs six checks against the extracted turn data. Each check
produces a status (\texttt{pass}, \texttt{warn}, or \texttt{fail}), a
confidence level, and a detail string.

\paragraph{Check 1: Zero-Tool Completion.}
Flags when the model's message contains completion indicators (multilingual:
English, Italian, French, Spanish, Portuguese, German) but the turn has zero
tool calls.

\begin{lstlisting}[language=bash,caption={Check 1: Zero-tool completion detection.}]
COMPLETION_RE='(done|complete[d]?|finished|fatto|completo|
  completato|terminat[oaie]*|tutti|all|every|each|ogni|
  termine|complete|terminado|completado|concluido|
  finalizado|fertig|abgeschlossen)'
MSG_LOWER=$(echo "$LAST_MSG" | tr '[:upper:]' '[:lower:]')
HAS_COMPLETION=$(echo "$MSG_LOWER" \
  | grep -ciP "$COMPLETION_RE" || true)

if (( HAS_COMPLETION > 0 )) && (( TOTAL_TOOLS == 0 )); then
  CK_zero_tool_s="fail"; CK_zero_tool_c="high"
  CK_zero_tool_d="Message contains completion indicators \
    but turn has 0 tool calls"
  HAS_FAIL=true
fi
\end{lstlisting}

\paragraph{Check 2: Context-Aware Numerical Discrepancy.}
Extracts numbers the model claims to have processed (e.g., ``analyzed 15
competitors'') and compares against the relevant tool call count. The check
is context-aware: if the message references screenshots, it counts image
reads; if it references pages, it counts \texttt{WebFetch} calls.

\begin{lstlisting}[language=bash,caption={Check 2: Numerical discrepancy (simplified).}]
WORK_NUMBERS=$(echo "$LAST_MSG" \
  | grep -oP '\b(\d+)\b\s*(competitor|screenshot|
    image|file|test|site|page)' \
  | grep -oP '^\d+' || true)

for NUM in $WORK_NUMBERS; do
  if (( NUM <= 5 )); then continue; fi
  RELEVANT_COUNT=$TOTAL_TOOLS  # default
  # Override based on context keywords:
  # screenshots -> IMAGE_READS
  # pages/sites -> WebFetch count
  # files -> non-image Read count
  # tests -> Bash calls with test keywords
  THRESHOLD_HIGH=$(( NUM * 50 / 100 ))
  THRESHOLD=$(( NUM * 85 / 100 ))
  if (( RELEVANT_COUNT < THRESHOLD_HIGH )); then
    CK_numerical_s="fail"; CK_numerical_c="high"
    HAS_FAIL=true; break
  elif (( RELEVANT_COUNT < THRESHOLD )); then
    CK_numerical_s="fail"; CK_numerical_c="medium"
    HAS_FAIL=true; break
  fi
done
\end{lstlisting}

\paragraph{Check 3: Test/Build Claim Without Execution.}
Detects when the model claims test or build results (``tests pass'',
``build succeeded'', ``0 errors'') but no Bash command in the turn
contains test or build keywords.

\begin{lstlisting}[language=bash,caption={Check 3: Test claim without execution.}]
TEST_CLAIM_RE='(tests?\s+pass|build\s+succeed|
  lint\s+clean|0\s+error|no\s+error)'
if echo "$MSG_LOWER" | grep -qiP "$TEST_CLAIM_RE" \
    && [[ "$HAS_TEST_CMD" == "false" ]]; then
  CK_test_claim_s="fail"; CK_test_claim_c="high"
  CK_test_claim_d="Message claims test/build results \
    but no test/build Bash commands found in turn"
  HAS_FAIL=true
fi
\end{lstlisting}

\paragraph{Check 4: Subagent Trust Without Verification.}
Warns when the model references subagent results as fact (``the agent
found\ldots'') but performed no \texttt{Read} or \texttt{Bash} calls
after the last \texttt{Agent} dispatch.

\begin{lstlisting}[language=bash,caption={Check 4: Subagent trust without verification.}]
AGENT_REF_RE="(agent found|agent completed|
  agent analysis|agent report)"
if echo "$MSG_LOWER" | grep -qiP "$AGENT_REF_RE" \
    && [[ "$HAS_AGENT_CALLS" == "true" ]] \
    && [[ "$VERIFIED_AFTER_AGENT" == "false" ]]; then
  CK_subagent_s="warn"; CK_subagent_c="medium"
  CK_subagent_d="Message references agent results \
    but no Read/Bash calls after last Agent dispatch"
  HAS_WARN=true
fi
\end{lstlisting}

\paragraph{Check 5: Completion Scope Mismatch.}
Warns when the model uses totality language (``all'', ``every'',
``complete'') with fewer than 5 tool calls and a message longer than
200 characters---a heuristic for verbose completion claims unsupported by
commensurate tool activity.

\begin{lstlisting}[language=bash,caption={Check 5: Scope mismatch.}]
TOTALITY_RE='(all|every|each|complete|tutti|ogni|
  tutto|completo|finished|done|tous|cada|alle|jeder)'
if echo "$MSG_LOWER" | grep -qiP "$TOTALITY_RE" \
    && (( TOTAL_TOOLS < 5 )) \
    && (( MSG_LEN > 200 )); then
  CK_scope_s="warn"; CK_scope_c="medium"
  HAS_WARN=true
fi
\end{lstlisting}

\paragraph{Check 6: VIBE\_GATE Verification.}
When a VIBE skill was invoked and the model claims completion, the sentinel
verifies that \texttt{VIBE\_GATE} markers exist in the Bash output. If markers
are present, it cross-references the marker values against numbers claimed in
the completion message.

\begin{lstlisting}[language=bash,caption={Check 6: VIBE\_GATE marker verification.}]
if [[ "$HAS_SKILL" == "true" ]] \
    && (( HAS_COMPLETION > 0 )); then
  if (( GATE_COUNT == 0 )); then
    CK_gate_s="fail"; CK_gate_c="high"
    CK_gate_d="VIBE skill used and completion claimed \
      but no VIBE_GATE markers found"
    HAS_FAIL=true
  else
    # Cross-reference marker values against
    # numbers in the message
    for i in $(seq 0 $((GATE_COUNT - 1))); do
      MARKER_KEY=$(echo "$GATE_MARKERS" \
        | jq -r ".[$i].key")
      MARKER_VAL=$(echo "$GATE_MARKERS" \
        | jq -r ".[$i].val")
      if ! echo "$MARKER_VAL" \
          | grep -qP '^\d+$'; then continue; fi
      for MSG_NUM in $MSG_NUMBERS; do
        if (( MSG_NUM > 5 )) \
            && (( MARKER_VAL < MSG_NUM )); then
          CK_gate_s="fail"; CK_gate_c="high"
          CK_gate_d="VIBE_GATE ${MARKER_KEY}=\
            ${MARKER_VAL} but message claims ${MSG_NUM}"
          HAS_FAIL=true
          break 2
        fi
      done
    done
  fi
fi
\end{lstlisting}

\subsection{Resolution Mode}

When the sentinel blocks a response (exit code~2), it sets a flag file and
enters \emph{resolution mode} for subsequent turns. In resolution mode, the
sentinel checks whether the model has performed new tool calls or provided
specific counts (e.g., ``I completed 14 of 20''). If so, the block is lifted.
If the model instead repeats totality claims without new evidence, the block
persists.

\begin{lstlisting}[language=bash,caption={Resolution mode handler.}]
if [[ "$RESOLUTION_MODE" == "true" ]]; then
  HAS_NEW_TOOLS=false
  if (( TOTAL_TOOLS > 0 )); then
    HAS_NEW_TOOLS=true
  fi
  HAS_SPECIFIC_COUNTS=false
  if echo "$LAST_MSG" \
      | grep -qP '\d+\s*(of|out of|di|su)\s*\d+'; then
    HAS_SPECIFIC_COUNTS=true
  fi
  if [[ "$HAS_NEW_TOOLS" == "true" ]] \
      || [[ "$HAS_SPECIFIC_COUNTS" == "true" ]]; then
    rm -f "$BLOCK_FLAG"
    exit 0  # resolved
  fi
  # Still claiming totality without evidence
  TOTALITY_RE='(all|every|each|complete|tutti|ogni|
    tutto|completo|finished|done|tous|cada|alle|jeder)'
  if echo "$MSG_LOWER" | grep -qiP "$TOTALITY_RE" \
      && (( TOTAL_TOOLS == 0 )); then
    cat >&2 << 'RESOLUTION_BLOCK'
VIBE INTEGRITY -- your previous response was flagged.
Your follow-up still claims total completion without
new tool calls or specific counts.
YOUR RESPONSE MUST CONTAIN:
- Specific counts: "I completed X of Y."
- OR new tool calls completing the missing work
RESOLUTION_BLOCK
    exit 2
  fi
  rm -f "$BLOCK_FLAG"
  exit 0
fi
\end{lstlisting}
